# AegisBroker Development Plan

> Last Updated: 2026-04-09  
> Version: v0.1.1-plan

## 多語一致性原則補充

為符合 AegisBroker 控制型系統在全球化營運中的一致性要求，未來所有介面開發與系統訊息傳遞必須遵守下列規範：

1. **所有使用者介面（UI）必須支援五語切換**
   - `zh-TW`（繁體中文）
   - `zh-CN`（簡體中文）
   - `en`（English）
   - `es`（Español）
   - `tr`（Türkçe）

2. **所有系統非圖形介面對外訊息也必須支援五語切換**
   - 包含但不限於：
     - API 回應中的 user-facing message
     - 通知訊息（Email / Webhook / 站內訊息）
     - 匯出報表中的文字欄位
     - 背景作業回傳給使用者的狀態/錯誤說明

3. **錯誤碼與理由碼維持穩定，文案才做語系切換**
   - `error_code`、`reason_code` 必須維持 machine-usable 穩定值。
   - 前端與外部訊息層僅針對顯示文案做本地化（i18n）。

4. **開發與測試要求**
   - 新增功能時，必須同時提供五語文案鍵值。
   - 任何變更若新增使用者可見文案，PR 需附上 i18n key 清單。
   - 測試至少覆蓋：預設語系回退、缺漏 key 防護、五語可切換。

5. **文件與部署要求**
   - `infra/env/*` 應保留語系相關設定（如 `DEFAULT_LOCALE`, `ENABLED_LOCALES`）。
   - 架構文件與 API 合約若新增 user-facing message 欄位，需明確說明語系策略。

## 前端控制台版型規範補充（ChatGPT-style）

為降低使用者學習成本，平台用戶介面在後續迭代中新增以下版型規範：

1. **整體結構**
   - 採用「左側導覽 + 右側主內容」雙欄布局。
   - 左側為功能選單與狀態摘要；右側為主要操作區與資料顯示區。

2. **左側欄位要求**
   - 選單置於左側中上區域，提供主要功能切換入口。
   - 使用者名稱與常用功能按鈕需固定於左下區域，便於快速操作。

3. **右側主內容要求**
   - 右側集中呈現設定、表單、結果輸出與操作回饋。
   - 重要結果需可讀、可複製、可回溯（例如統一輸出區）。

4. **狀態同步要求**
   - 介面需提供與使用者操作相關的同步狀態（如 backend / auth / session）。
   - 狀態呈現需即時更新，且必須具備可視化標記（例如 badge/pill 狀態）。

5. **視覺色彩**
   - 主色調採冷色系（藍/青為主），保持控制台一致感與可讀性。
   - 對比需兼顧長時間操作的辨識度與可用性。

6. **實作原則**
   - 不得因版型調整而破壞既有 API 行為、事件追蹤與審計相關功能。
   - 版型變更需保留既有 i18n 能力與錯誤可觀測性。

7. **用戶狀態同步與系統紅綠燈（新增）**
   - 用戶狀態同步需支援登入/登出切換，並提供顯式操作入口（登入、登出按鈕）。
   - 左側狀態區需反映「是否已登入」與「是否啟用操作」的即時狀態，避免僅依 token 欄位隱性判斷。
   - 內容區右上需固定放置三個主要系統狀態指示（參考 Chrome 右上擴充功能群組的可掃視性）：
     1. `backend 狀態`
     2. `資料庫狀態`
     3. `外部交易所狀態`
   - 三項狀態需使用紅/黃/綠燈圖示排列顯示，並支援滑鼠 hover popup 顯示狀態文字簡述。
   - 若外部交易所狀態尚未接入健康檢查，也必須明確標示為「未知/待同步」，不可呈現為誤導性的綠燈。
